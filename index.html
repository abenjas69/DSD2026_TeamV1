TEAM CONTEXT

Why V1 matters

V1 is the analytical core of the project. It receives structured motion data, prepares it for inference, integrates the models, compares outputs against expected values, and produces results that other teams can use for APIs, dashboards, patient feedback, and clinical interpretation.

Input

Motion data arrives from the Sensor area through the Sensor → Server contract.

Processing

V1 handles preprocessing, feature preparation, model execution, and output shaping.

Output

Recognition results and angle comparison outputs are prepared for V2 and later Monitor use.

SYSTEM ARCHITECTURE

Where V1 sits in the system

The system is divided into Sensor, Server, and Monitor. V1 belongs to the Server area together with V2, not as a separate layer.

Sensor Layer

S1 + S2

Firmware, wireless communication, data acquisition, and mock sensor data generation.

Server Layer

V1 + V2

V1 · AI & Motion Recognition

Preprocessing, inference, model integration, angle comparison, validation.

V2 · API & Storage

REST/WebSocket API, persistence, recommendations, and broader integration.

Monitor Layer

M1 + M2

Patient-facing mobile application and clinical web dashboard.

CORE DELIVERABLES

What V1 will deliver

Model Integration

Integrate the motion recognition models into a usable server-side workflow.

Preprocessing

Prepare incoming sensor data for inference through validation and feature preparation.

Inference Pipeline

Build the end-to-end flow from motion data input to structured AI output.

Angle Comparison

Compare recognised joint angles against expected recovery values.

V1–V2 Internal Contract

Define the internal contract and output format between both Server teams.

Validation & Benchmarking

Evaluate reliability, metrics, assumptions, and limitations of the produced outputs.
