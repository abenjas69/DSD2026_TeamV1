0. Background

This document describes the requirement analysis for Team V1 in the Limb Motion Recognition and Assistant project. The project is developed in a distributed setting between UTAD and Jilin University.

Inside the project, the Server area receives motion data from the Sensor area, applies AI-based motion recognition, stores session information, and exposes processed results to the Monitor area.

Team V1 is responsible for the AI and motion-recognition part of the Server pipeline. V1 is not responsible for the physical sensors and is not responsible for the final patient or clinical interfaces.

0.1 Purpose of this document

  • To define the requirement analysis of Team V1 using the shared SRS template.
  • To make clear what Team V1 expects to receive from other teams.
  • To make clear what Team V1 will provide to other teams.
  • To define the main V1 interfaces, use cases, and output-delivery approach.

0.2 Scope of Team V1

  • Validate and preprocess incoming motion data so it becomes model-ready input.
  • Run the motion-recognition inference pipeline.
  • Compare recognised joint angles with expected recovery values available to the server workflow.
  • Build structured AI results and provide them to Team V2 through an internal server contract.
  • Keep useful processing metadata, warnings, and error states with each result.

0.3 Team boundaries

  • The Sensor teams define and deliver the data stream from hardware to server. Team S2 co-owns IF1 with V1.
  • Team V1 focuses on preprocessing, inference, angle comparison, result packaging, and the V1-V2 internal contract.
  • Team V2 focuses on API gateway, storage, authentication, recommendation logic, and IF2 with the Monitor teams.

0.4.3 Interface definition summary

Interface Direction Owners Main Content
IF1 S2 -> V1 S2 and V1 Sensor-session payload structure, field meaning, sampling rules, status flags, and error handling.

0.4.4 Output delivery approach

  • V1 will package its output as a structured internal result object that follows the agreed V1-V2 contract.
  • Each result should include the processing status so V2 can handle both success and failure cases in a consistent way.
  • When available and useful, the result should also include confidence, model version, timestamp, and warning information.
  • The exact technical delivery mechanism can be synchronous or asynchronous, but the payload format must stay stable and clearly documented.

2.1.2 Basic Flow

Actor System (V1 Module)
S2 sends a new motion-data payload to the server. Receive the payload and start the processing routine.
Check whether the payload follows the expected IF1 structure.
Validate required fields and basic input status.
Prepare the payload for model-ready input.
Run the motion-recognition model on the prepared input.
Create a structured result with recognised motion output and processing status.
V2 can consume the result in the next internal step.

2.1.3 Alternative Flow

Actor System (V1 Module)
Payload does not match the expected IF1 format. Return a validation error and stop normal processing.
Required fields are missing or invalid. Return input-status information and mark the session as not processed.
The model or inference service is not available. Return a processing-failure status and log the issue for technical review.
Processing completes but confidence is low. Return a low-confidence status or warning together with result metadata.

2.2.2 Basic Flow

Actor System (V1 Module)
No direct external actor. The step starts inside V1 after successful inference. Read the recognised angle values and related metadata.
Load the expected recovery values or comparison reference used by the workflow.
Compare recognised values against expected values.
Create a structured comparison result with status, differences, and any relevant warning.
The richer output becomes available for the next V1-V2 handoff.

2.2.3 Alternative Flow

Actor System (V1 Module)
Reference values are not available. Return a partial result and mark the comparison step as incomplete.
Recognised angle output is incomplete. Return a warning and do not compute the normal comparison result.
The comparison rule needs clarification. Keep the rule marked as assumption-dependent until the project alignment is finished.

3.3 Delivery rules

  • The V1 output should be delivered in a structured internal format agreed with V2.
  • The payload should be easy to validate, store, and expose later through V2 services.
  • Normal and failed processing cases should use the same general contract structure so downstream handling stays consistent.
  • Field names, allowed status values, and required/optional metadata should be documented jointly by V1 and V2.

4. Functional Requirements Summary

ID Requirement Notes
FR1 The V1 module should receive motion data from the Sensor area through IF1. Input comes from S2 through the agreed Sensor -> Server contract.
FR2 The V1 module should check required fields and basic validity before inference starts. Examples: structure, missing values, status flags.
FR3 The V1 module should transform valid incoming data into model-ready input. Preprocessing and feature preparation.
FR4 The V1 module should run inference and generate structured motion-recognition output. Main AI processing step.
FR5 The V1 module should compare recognised joint angles with expected recovery values available to the workflow. Comparison result may be partial if reference values are not available.
FR6 The V1 module should provide structured results to V2 using a clear internal contract. Result schema, status, warnings, and metadata.
FR7 The V1 module should report processing errors or low-confidence outcomes in a consistent way. Important for traceability and later UI behaviour.
FR8 The V1 module should keep useful model and processing metadata with each result when relevant. Examples: model version, timestamp, confidence.
FR9 The V1 module should support validation and benchmarking-oriented records for technical review. Metrics and assumptions should be documented.

5. Non-Functional Requirements Summary

ID Requirement Notes
NFR1 Interface clarity Input and output formats should be documented in a way that other teams can understand quickly.
NFR2 Maintainability The pipeline should be organised so preprocessing changes, model updates, and contract changes are manageable.
NFR3 Testability The module should support peer review, cross-testing, and repeatable checks on sample data.
NFR4 Traceability Processing status, assumptions, warnings, and technical limitations should be easy to review later.
NFR5 Team readability Documentation and interface descriptions should use simple English and avoid unnecessary complexity.

6. Assumptions and Alignment Notes

  • IF1 should be frozen early enough to unblock stable V1 development.
  • The source of expected recovery values should be agreed by the server workflow before comparison logic is treated as fully fixed.
  • The V1-V2 internal contract should define required fields, optional fields, and allowed status values.
  • The exact technical delivery mechanism may still evolve, but the payload structure should remain stable once agreed.
